# ClickUp API Patterns Reference

Canonical patterns for all ClickUp API interactions across the plugin. ALL skills and protocols reference this file instead of duplicating patterns.

---

## Token Extraction (Canonical)

**Claude Code loads `env` from settings.json automatically at session start.**
`$CLICKUP_API_TOKEN` is already set — no bash extraction needed.

**Validation (single-statement, auto-approved):**
```bash
printenv CLICKUP_API_TOKEN
```

If empty: use the **Read TOOL** to read `SETTINGS_PATH`, extract `env.CLICKUP_API_TOKEN` from JSON.
**NEVER use python3 -c multi-line bash for token extraction** — multi-line is blocked by hooks.

**Validate format (2 separate single-statement calls):**
```bash
echo "$CLICKUP_API_TOKEN" | grep -q "^pk_" && echo "VALID" || echo "INVALID"
```

**Test token (single-statement curl — auto-approved by clickup-auto-approve.sh):**
```bash
RESPONSE=$(curl -s -X GET -H "Authorization: $CLICKUP_API_TOKEN" "https://api.clickup.com/api/v2/team")
```
```bash
echo "$RESPONSE" | grep -o '"name":"[^"]*"' | head -1 | cut -d'"' -f4
```

Execute as **two separate Bash calls** — variables don't persist, but the second call processes `$RESPONSE` inline.

## Hook Compatibility Rules (MANDATORY)

These rules apply to ALL bash commands generated by ALL agents:

| Rule | Reason |
|------|--------|
| Use `$VARIABLE` never `${VARIABLE}` | Claude Code has built-in `${}` detection that prompts user |
| Never `${VAR:n:m}` or `${#VAR}` | Bash parameter expansions with `{}` also trigger the prompt |
| Never `&&`, `\|\|`, `;` between statements | `bash-safe-auto-approve.sh` deny list — falls through to manual |
| Never multi-line commands (newlines) | Blocked by multi-line deny in `bash-safe-auto-approve.sh` |
| Never heredocs `<< 'EOF'` | Blocked by heredoc deny |
| Never `find \| while read` pipelines | Multi-line, complex — use Glob+Read tools instead |
| Always explicit `-X GET/POST/PUT/DELETE` | Required by `clickup-auto-approve.sh` to match permissions |
| Never `/tmp/` paths | macOS `/tmp/` → `/private/tmp/` symlink causes "outside working dir" |
| For local files: use Glob/Read/Edit tools | Not Bash — tools are pre-approved, bash requires hook matching |

**Single-statement curl is auto-approved** by `clickup-auto-approve.sh` when METHOD+URL matches `permissions.allow`.
**Multi-statement scripts** are also approved if they contain `api.clickup.com` with explicit `-X METHOD`.

## Response Handling

### NEVER use jq on ClickUp responses

ClickUp API responses contain control characters (U+0000-U+001F) that **break both `jq` and Python `json.loads()`**. This is a known ClickUp bug, not fixable on our side.

```bash
# WRONG — will fail with parse errors
curl ... | jq '.id'

# WRONG — will fail with json.decoder.JSONDecodeError
curl ... | python3 -c "import json,sys; print(json.loads(sys.stdin.read())['id'])"

# CORRECT — always capture first, extract with grep
RESPONSE=$(curl -s -X POST ...)
TASK_ID=$(echo "$RESPONSE" | grep -o '"id":"[^"]*"' | head -1 | cut -d'"' -f4)
```

### Common extraction patterns

```bash
# Extract task ID
TASK_ID=$(echo "$RESPONSE" | grep -o '"id":"[^"]*"' | head -1 | cut -d'"' -f4)

# Extract task name
TASK_NAME=$(echo "$RESPONSE" | grep -o '"name":"[^"]*"' | head -1 | cut -d'"' -f4)

# Detect errors
ERROR=$(echo "$RESPONSE" | grep -o '"err":"[^"]*"' | head -1 | cut -d'"' -f4)
[ -n "$ERROR" ] && echo "API Error: $ERROR"

# Extract multiple IDs (e.g., from subtask list)
echo "$RESPONSE" | grep -o '"id":"[^"]*"' | cut -d'"' -f4
```

### jq exception: non-ClickUp JSON

`jq` is safe for parsing Claude Code's own JSON (settings.json, tool input). **Only ClickUp API responses** contain the problematic control characters.

```bash
# SAFE — settings.json is our own file
jq -r '.permissions.allow[]' "$SETTINGS_PATH"

# SAFE — Claude Code hook input is clean JSON
echo "$input" | jq -r '.tool_input.command // ""'
```

## Description Handling

### Read description (markdown)

**ALWAYS** use `markdown_description` field + `include_markdown_description=true` parameter:

```bash
RESPONSE=$(curl -s -X GET -H "Authorization: $CLICKUP_API_TOKEN" \
  "https://api.clickup.com/api/v2/task/$TASK_ID?include_markdown_description=true")
```

Extract with sanitization (remove control chars first):

```bash
CURRENT_DESC=$(echo "$RESPONSE" | tr -d '\000-\011\013-\037' | python3 -c "
import sys, json
data = json.loads(sys.stdin.read())
print(data.get('markdown_description', '') or '')
")
```

### Write description (markdown)

**ALWAYS** use the `markdown_description` field in PUT, **NEVER** the `description` field:

```bash
# Build payload — use heredoc for complex markdown
JSON_PAYLOAD=$(python3 -c "
import json, sys
desc = sys.stdin.read()
print(json.dumps({'markdown_description': desc}))
" <<'DESCEOF'
YOUR MARKDOWN CONTENT HERE
DESCEOF
)

curl -s -X PUT -H "Authorization: $CLICKUP_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d "$JSON_PAYLOAD" \
  "https://api.clickup.com/api/v2/task/$TASK_ID"
```

### Key rules

| Rule | Details |
|------|---------|
| GET description | Use `?include_markdown_description=true`, read `markdown_description` field |
| PUT description | Use `markdown_description` field, NEVER `description` |
| Never overwrite | For updates: read from local cache (`.claude/code-reviews/`), modify locally, then PUT. For creation: use 2-step (POST name → PUT full description composed locally). Only GET from ClickUp as fallback if local file doesn't exist. |
| HTML comments | `<!-- -->` render as VISIBLE TEXT in ClickUp. Never use as delimiters. |
| Markdown checklists | `- [ ]` renders as interactive checkboxes in ClickUp. Zero API calls. |

## Task Creation (2-Step: POST name → PUT full description)

Create a ticket in 2 steps: POST with name only (get task_id), then PUT with COMPLETE description composed locally. This avoids the round-trip GET that caused description wipes in v3.1.0.

**Step 1 — POST: Create task with name only**

```bash
JSON_PAYLOAD=$(python3 -c "
import json
print(json.dumps({
    'name': '$TASK_NAME',
    'parent': '$PARENT_TASK_ID',
    'priority': $PRIORITY_INT
}))
")

RESPONSE=$(curl -s -X POST \
  -H "Authorization: $CLICKUP_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d "$JSON_PAYLOAD" \
  "https://api.clickup.com/api/v2/list/$LIST_ID/task")

TASK_ID=$(echo "$RESPONSE" | grep -o '"id":"[^"]*"' | head -1 | cut -d'"' -f4)
```

**Step 2 — PUT: Write COMPLETE description (composed locally)**

```bash
JSON_PAYLOAD=$(python3 -c "
import json, sys
desc = sys.stdin.read()
print(json.dumps({'markdown_description': desc}))
" <<'DESCEOF'
YOUR COMPLETE MARKDOWN DESCRIPTION HERE
(composed locally — never read back from ClickUp)
DESCEOF
)

curl -s -X PUT -H "Authorization: $CLICKUP_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d "$JSON_PAYLOAD" \
  "https://api.clickup.com/api/v2/task/$TASK_ID"
```

**WHY 2 steps:** The description needs the real `task_id` (e.g., for Issue Name block). By composing locally and never reading back from ClickUp, we eliminate the GET round-trip that loses markdown formatting.

**Total API per ticket:** 2 calls (1 POST + 1 PUT). Down from ~12 in v3.1.0.

**Checklists as markdown:** `- [ ]` items render as interactive checkboxes in ClickUp. Zero API calls.

**POST fields:** `name`, `parent`, `priority`
**PUT fields:** `markdown_description`
**NEVER include:** `tags`, `assignees`, `custom_fields`, `due_date`, `start_date`

## Status Updates

### Case-mapping (MANDATORY before any status update)

```bash
RESPONSE=$(curl -s -X GET -H "Authorization: $CLICKUP_API_TOKEN" \
  "https://api.clickup.com/api/v2/list/$LIST_ID")

# Extract actual status names
echo "$RESPONSE" | grep -oE '"status":"[^"]*"' | cut -d'"' -f4 | sort -u
```

Build mapping dictionary and use mapped values:

```bash
# Use the EXACT case from the API response
curl -s -X PUT -H "Authorization: $CLICKUP_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"status\":\"$MAPPED_STATUS\"}" \
  "https://api.clickup.com/api/v2/task/$TASK_ID"
```

### Status flow per skill

| Skill | Flow |
|-------|------|
| Audit | Creates tickets at `open` status (ClickUp default) |
| Planning | `open` -> `planning` -> `ready for dev` (or `Closed` if invalid) |
| Fix | `ready for dev` -> `in progress` -> `code review` -> `testing` -> `deploy to staging` |

**NEVER batch status updates. Update IMMEDIATELY per action.**

## Rate Limiting

### Proactive counter

```bash
RATE_FILE="{REVIEW_DIR}/progress/rate-limit.log"

# Before EACH API call:
RECENT=$(tail -80 "$RATE_FILE" 2>/dev/null | \
  awk -v cutoff="$(date -v-60S +%s 2>/dev/null || date -d '60 seconds ago' +%s)" \
  '$1 >= cutoff' | wc -l)

if [ "$RECENT" -ge 80 ]; then
  echo "Rate limit approaching ($RECENT/min). Sleeping 20s..."
  sleep 20
fi
echo "$(date +%s)" >> "$RATE_FILE"
```

**Thresholds:**
- 80 requests/min: proactive sleep (20 seconds)
- 429 received: wait 60 seconds, retry once
- Still 429: stop, report to user

## Error Handling

```bash
# Standard error check after any API call
RESPONSE=$(curl -s -o /dev/null -w "%{http_code}" -X METHOD ...)
if [ "$RESPONSE" = "429" ]; then
  echo "Rate limited. Waiting 60s..."
  sleep 60
  # Retry once
fi

# Or check response body for errors
ERROR=$(echo "$RESPONSE" | grep -o '"err":"[^"]*"' | head -1 | cut -d'"' -f4)
if [ -n "$ERROR" ]; then
  echo "API Error: $ERROR"
  # Log and continue with remaining operations
fi
```

### Retry policy

| Scenario | Action |
|----------|--------|
| HTTP 429 | Wait 60s, retry once |
| HTTP 500 | Wait 2s, retry once |
| Network error | Wait 2s, retry once |
| Invalid response | Log, skip, continue |
| Auth error (401) | Stop execution, report token issue |

## Payload Discipline

The auto-approve hook checks METHOD + URL only — it does NOT inspect the JSON body. Only use authorized fields.

**Allowed fields in POST /list/{id}/task:**
- `name`, `markdown_description`, `parent`, `priority`

**Allowed fields in PUT /task/{id}:**
- `markdown_description`, `status`, `priority`

**NEVER include:** `tags`, `assignees`, `custom_fields`, `due_date`, `start_date`

## Comments Endpoint

### Fetch comments (includes nested replies)

```bash
RESPONSE=$(curl -s -X GET -H "Authorization: $CLICKUP_API_TOKEN" \
  "https://api.clickup.com/api/v2/task/$TASK_ID/comment")
```

**Response structure:** Each comment object contains `id`, `comment_text`, `user.username`, `date` (Unix ms), and a `reply_comments` array with nested replies (same fields).

### Extract latest comment/reply ID

```bash
# Get all comment IDs (top-level)
COMMENT_IDS=$(echo "$RESPONSE" | grep -o '"id":"[^"]*"' | cut -d'"' -f4)

# Get the most recent comment ID (first in response — ClickUp returns newest first)
LATEST_COMMENT_ID=$(echo "$RESPONSE" | grep -o '"id":"[^"]*"' | head -1 | cut -d'"' -f4)
```

### Detect new comments

Compare latest comment/reply ID against `last_comment_id` from local cache frontmatter:
- **Different** → new comments exist, process them
- **Same** → no new activity since last sync

### Comment permission

Uses the same `GET /task/*` permission pattern already configured in the setup wizard. No additional permissions needed.

## Local File Cache

### Purpose

Local `.md` files in `.claude/code-reviews/` serve as:
1. **Composition surface** — write content locally, then PUT to ClickUp
2. **Fallback** — if PUT fails or session dies, content is preserved on disk
3. **Persistent state** — YAML frontmatter tracks sync state across sessions
4. **Comment detection** — `last_comment_id` enables detecting human feedback between phases

**ClickUp remains the SOT.** The local cache mirrors it and is the workspace for building content before PUT.

### Folder structure

```
.claude/code-reviews/
└── {main_task_id} - {Review Name}/
    ├── _main.md
    ├── {area_task_id} - {Area Name}/
    │   ├── _area.md
    │   ├── {finding_task_id}.md
    │   └── {finding_task_id}.md
    ├── {area_task_id} - {Area Name}/
    │   └── _area.md
    └── ...
```

**Naming:** `{task_id} - {Name}` for folders (main + areas). `{task_id}.md` for finding files.

### YAML Frontmatter (every `.md` file)

```yaml
---
task_id: 86c8j7k3a
area: Security
severity: High
priority: 2
status: open
last_synced: 2026-03-08T14:30:00
last_comment_id: ""
---
```

Fields:
- `task_id` — ClickUp task ID
- `area` — category name (finding files only)
- `severity` / `priority` — from finding (finding files only)
- `status` — mirrors ClickUp status
- `last_synced` — ISO timestamp of last PUT to ClickUp
- `last_comment_id` — ID of last processed comment (empty string if none)

### Gitignore

**MANDATORY:** Maestro adds `.claude/code-reviews/` to `.gitignore` at start of each skill if not present.

```bash
if ! grep -q 'code-reviews/' .gitignore 2>/dev/null; then
  echo '.claude/code-reviews/' >> .gitignore
fi
```

### File content format

Finding `.md` files contain YAML frontmatter (`---` delimited) followed by the ticket description. The description is composed/enriched locally, then PUT to ClickUp.

**CRITICAL — Strip frontmatter before PUT:** YAML frontmatter is LOCAL-ONLY metadata (task_id, area, status, last_synced, etc.). It must NEVER be sent to ClickUp. When PUTting to ClickUp, always extract content AFTER the second `---` delimiter:

```bash
# Correct: n>=2 skips everything between first and second ---
DESC_CONTENT=$(awk '/^---$/{n++; next} n>=2' "$FINDING_FILE")

# WRONG: n>=1 includes the frontmatter content itself
# DESC_CONTENT=$(awk '/^---$/{n++; next} n>=1' "$FINDING_FILE")
```

This applies to ALL PUTs: findings, area descriptions, main task descriptions.

### Sync protocol

1. **Create:** Write `.md` file first → PUT to ClickUp → update `last_synced` in frontmatter
2. **Enrich (planning):** Read local `.md` → enrich content → write back → PUT to ClickUp → update frontmatter
3. **Read comments:** Fetch via API → compare `last_comment_id` → process new ones → update frontmatter
4. **Never GET description from ClickUp to modify it.** The local file IS the composition surface.
