# ClickUp Code Review Plugin — Setup Wizard & API Directa

**Data:** 2026-03-03
**Versão plugin:** 2.0.0 (upgrade de 1.0.0)
**Escopo:** Adicionar setup wizard interactivo + migrar de MCP para API directa

---

## Motivação

O ClickUp MCP desconecta a cada ~60-70 calls (OAuth session limits). A API directa com token `pk_` é stateless — cada request é independente, sem sessão que possa expirar. Adicionalmente, o plugin não tinha onboarding — o utilizador precisava de configurar manualmente MEMORY.md e credentials.

## Decisões de Design

### 1. Dois Entry Points

| Slash command | Quando | O que faz |
|---|---|---|
| `/clickup-code-review` | Execução normal | Fase 0 verifica config → se OK, avança. Se falta algo, corre setup para o que falta |
| `/clickup-code-review:setup` | Reconfiguração | Wizard completo: token, workspace, list, shortname, permissões |

### 2. Onde Vive Cada Configuração

| Dado | Onde | Escopo |
|---|---|---|
| API Token (`CLICKUP_API_TOKEN`) | `settings.json` → `env` | Global |
| Permissões pré-autorizadas | `settings.json` → `permissions.allow` | Global |
| Workspace ID | MEMORY.md do projecto | Per-project |
| List ID | MEMORY.md do projecto | Per-project |
| Shortname | MEMORY.md do projecto | Per-project |

### 3. Auto-detecção do settings.json

1. Verificar `$PWD/.claude/settings.json` → tem `clickup-code-review` no `enabledPlugins`?
2. Verificar `$HOME/.claude/settings.json` → tem?
3. Se nenhum → perguntar ao utilizador

Usa `$HOME` para compatibilidade cross-OS (macOS, Linux, Windows).

### 4. API Directa em vez de MCP

Todas as chamadas ao ClickUp passam a ser `curl` com `-X METHOD` explícito:

```bash
curl -s -X POST \
  -H "Authorization: $CLICKUP_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"...","markdown_description":"..."}' \
  "https://api.clickup.com/api/v2/list/{list_id}/task"
```

O método explícito (`-X GET`, `-X POST`, `-X PUT`, `-X DELETE`) permite permissões granulares no settings.json.

O utilizador escolhe individualmente quais operações autorizar, em 3 fases.

### Fase 3 — DELETE (2 operações → 1 permissão)

| # | Operação | Endpoint | Permissão Bash |
|---|----------|----------|----------------|
| D1 | Remover tag | DELETE `/task/{id}/tag/{name}` | `Bash(curl:*-X DELETE*api.clickup.com/api/v2/task/*)` |

### Operações Necessárias vs Opcionais

| Operação | Necessidade | Porquê |
|----------|-------------|--------|
| R1-R2 (workspaces/spaces) | **Necessária** (setup) | Navegar hierarquia para escolher list |
| R5-R7 (folders/lists) | **Necessária** (setup) | Navegar até à list de destino |
| W1 (criar task) | **Necessária** (core) | Criar tickets de review |
| W2 (actualizar task) | **Necessária** (core) | Actualizar descrição com stats |
| Todas as outras | Opcional | Funcionalidades extra |

Se o utilizador não autorizar uma necessária, o wizard avisa mas permite continuar — o bot pede permissão manual em cada execução.

### Apresentação com AskUserQuestion Paginado

Cada fase usa AskUserQuestion com multiSelect. Operações agrupadas em blocos de 2-4 por pergunta, com opção "Todas" como atalho na primeira pergunta de cada fase. Múltiplas calls para cobrir todas as operações.

## Setup Wizard — 5 Passos

### Passo 1: Detectar settings.json

Verificar `$PWD/.claude/settings.json` → `$HOME/.claude/settings.json` → perguntar.
Critério: ficheiro tem `clickup-code-review` no `enabledPlugins`.

### Passo 2: API Token

1. Verificar `$CLICKUP_API_TOKEN` no ambiente
2. Se não: verificar `settings.json` → `env`
3. Se não: pedir ao utilizador (link para https://app.clickup.com/settings/apps)
4. Validar: começa com `pk_`
5. Testar: `curl GET /team` → sucesso?
6. Guardar em `settings.json` → `env` → `CLICKUP_API_TOKEN`

### Passo 3: Workspace → Space → Folder → List

Navegação interactiva via API:
1. `GET /team` → AskUserQuestion "Qual workspace?"
2. `GET /team/{id}/space` → "Qual space?"
3. `GET /space/{id}/folder` → "Qual folder?"
4. `GET /folder/{id}/list` → "Em que list criar tickets?"

Se já configurado: mostra valores actuais, pergunta "Manter ou alterar?"
Guardar workspace_id, list_id no MEMORY.md.

### Passo 4: Shortname

Se não existe no MEMORY.md: "Qual a slug? (ex: FSL, HT2, CDM)"
Se já existe: "Shortname actual: FSL. Manter ou alterar?"
Guardar no MEMORY.md.

### Passo 5: Permissões (Read/Write/Delete)

3 fases com AskUserQuestion paginado (ver secção Arquitectura de Permissões).
Merge no array `permissions.allow` do settings.json — não apaga existentes.

---

## Fase 0 no SKILL.md Principal

Antes de criar team ou lançar agentes:

1. Ler MEMORY.md → extrair workspace_id, list_id, shortname
2. Verificar token ($CLICKUP_API_TOKEN)
3. Se TUDO presente → avançar para review
4. Se algo em falta → correr setup APENAS para o que falta
5. Nunca bloquear — se o utilizador recusar setup, avisar e pedir permissão manual durante execução
