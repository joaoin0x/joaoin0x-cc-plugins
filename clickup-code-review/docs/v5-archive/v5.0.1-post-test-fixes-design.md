# ClickUp Code Review Plugin v5.0.1 — Post-Test Fixes Design

**Date:** 2026-03-13
**Context:** 1st test of v5.0 on `codewebsite84` revealed systemic orchestration issues. Result was correct (62 tickets, 0 duplicates), but 3 context overflows in ~50 minutes, total loss of agent state, and user couldn't leave the keyboard.

## Root Cause Analysis (from transcript)

The Maestro became the bottleneck because:
1. **Specialists sent findings TO Maestro** (not to DA) — architectural violation
2. **Maestro did relay/batching** — sent 14,444-char batch of 43 findings to DA
3. **Echo-back protocol** — every SendMessage echoed back with analysis (~30-50k tokens per round-trip)
4. **DA verdicts included full findings** (~3,000 chars each) — accumulated in Maestro context

## Architecture Change: File-Based Communication

### Core Principle
SendMessage carries only **notifications** (~60 chars). Finding content lives in **files** (/tmp/findings/).

### Complete Workflow

#### PHASE 0: Config
- Maestro spawns CU Manager
- CU Manager verifies config (token, list_id, shortname, status mapping)
- CU Manager → Maestro: READY + config data
- Maestro presents AskUserQuestion for category selection

#### PHASE 1: Spawn (MANDATORY ORDER)
1. Maestro → CU Manager: "Create main task + area subtasks"
2. CU Manager creates on ClickUp (POST) → receives task_ids
3. CU Manager creates ENTIRE local folder tree at once:
   - `.claude/code-reviews/{review_dir}/` + all area subdirs
   - `/tmp/findings/` directory
4. CU Manager → Maestro: all task_ids
5. Maestro spawns DA with MODE: FINDING-FILTER + area_task_ids
6. DA → Maestro: READY
7. Maestro spawns Specialists (all selected categories)

#### PHASE 2: Parallel Audit
- Specialist analyzes code
- Specialist writes finding to `/tmp/findings/{specialist}-{n}.md`
- Specialist → DA (SendMessage): "Valida `/tmp/findings/security-1.md`"

#### PHASE 3: DA Filtering
- DA reads /tmp file
- **Obvious →** DA writes verdict to file + DA → Maestro: "APPROVED /tmp/findings/security-1.md" (~60 chars)
- **Doubt →** DA ↔ Specialist debate via SendMessage (fast, max 3 rounds). After consensus, DA writes verdict + notifies Maestro
- **Deadlock →** Maestro reads /tmp file + mediates via SendMessage

#### PHASE 4: Ticket Creation (streaming, 1 at a time)
- Maestro → CU Manager: "Create ticket from `/tmp/findings/security-1.md`"
- CU Manager:
  1. Reads /tmp file
  2. POST name to ClickUp → receives task_id
  3. Creates `.claude/code-reviews/{review}/{area}/{task_id}.md` (copies /tmp content + adds YAML frontmatter)
  4. PUT description to ClickUp
  5. → Maestro: "task_id created" (~20 chars)

#### PHASE 5: Summary
- CU Manager queries ClickUp for real ticket data
- Maestro presents data-driven summary to user

### Maestro Context Per Finding
- **Before (v5.0):** ~3,000 chars (full finding in verdict)
- **After (v5.0.1):** ~80 chars (path + verdict + task_id)
- **62 findings:** ~5KB vs ~186KB

## Changes By File

### 1. skills/audit/SKILL.md — Major rewrite of Phases 0-3

- **Spawn order explicit:** CU Manager → DA → Specialists (MANDATORY)
- **FORBIDDEN section:** Maestro NEVER receives/relays/batches findings
- **ASCII diagram:** Visual flow showing who talks to whom
- **Spawn template:** Each specialist gets "/tmp + DA as recipient" as instruction #1
- **File-based finding flow:** /tmp/findings/ as communication medium
- **Recovery protocol:** Read /tmp + .claude/code-reviews/, verify agents alive via SendMessage BEFORE re-spawning
- **Streaming rule:** Maestro sends 1 finding at a time to CU Manager (NEVER batch)
- **grep -P FORBIDDEN:** macOS incompatible, use grep -E

### 2. agents/clickup-manager.md — /tmp reading + local tree creation

- **Phase 1 duty:** Create ClickUp hierarchy + entire local folder tree + /tmp/findings/ in one go
- **Ticket creation:** Read from /tmp file instead of receiving content via SendMessage
- **Copy /tmp → .claude/code-reviews/:** Add frontmatter, rename to task_id.md
- **Dedup:** Exact match on file:line or title → SKIP (no fuzzy %)

### 3. agents/devils-advocate.md — File-based + batch handling + keepalive

- **Read findings from /tmp files** instead of receiving full content via SendMessage
- **Write verdict to /tmp file** (append section at bottom)
- **Send short verdict to Maestro** (~60 chars, path only)
- **Batch handling:** If multiple findings arrive at once, process individually
- **Keepalive response:** Brief status when pinged

### 4. hooks/clickup-auto-approve.sh — Local file ops + grep fix

- **Add pattern:** Approve cat/tee/mkdir for `.claude/code-reviews/` and `/tmp/findings/`
- **No grep -P anywhere** (already clean, but verify)

### 5. Agent colors — Standardize

- devils-advocate: red (keep)
- clickup-manager: blue (change from gray)
- quality-specialist: yellow (change from orange)
- investigation-specialist: cyan (change from gray)
- README.md: align with agent files

### 6. Legacy cleanup (DONE)

- `skills/clickup-code-review/SKILL.md` stripped to 2-line redirect
- `skills/clickup-code-review/agent-prompts/` deleted
- `skills/planning/agent-prompts/` deleted
- Archived to `docs/v4-archive/`
