# ClickUp Code Review Plugin v5.0 — Redesign Completo

## Context

O plugin ClickUp Code Review (v4.1.2) tem uma falha de design fundamental: **o pipeline baseia-se em confianca nos reports dos agentes sem verificacao de evidencia**. O Maestro muda status porque um agente *disse* que fez algo, nao porque *provou* que fez.

**Prova concreta:** O ticket `86c8j5q7b` (getStatusLabel) passou por TODAS as fases (open -> planning -> ready for dev -> in progress -> testing -> deploy to staging) sem um unico commit. Zero codigo escrito. Status falso.

### Problemas Identificados

1. **Zero verificacao de evidencia** — nenhuma transicao de status exige prova (commit SHA, test results)
2. **Sem binding commit <-> ticket** — commit message tem ticket ID, mas o ticket nao tem commit SHA
3. **Agentes dependem de tipos built-in Anthropic** — se Anthropic mudar system prompts, plugin quebra
4. **Testing superficial** — QA agents so carregavam paginas em vez de interaccoes reais
5. **Sem status RE-OPEN** — tickets que falham testing ficam "in progress" sem distincao
6. **Maestro faz demasiado** — orquestra, gere ClickUp, toma decisoes, comunica com user

### Objectivo

Redesenhar o plugin para v5.0 com:
- Evidence gates em cada transicao de status
- Todos os agentes internalizados (zero dependencia de tipos built-in)
- Novo agente ClickUp Manager (separa gestao de tickets da orquestracao)
- Novo skill `:testing` para testes funcionais browser
- Rename `:clickup-code-review` -> `:audit`
- Commit SHA + Branch na descricao de cada ticket
- Agentes persistentes (shutdown so por instrucao do Maestro + user)
- Chrome DevTools MCP obrigatorio para testes
- Regra de seguranca: zero credenciais reais em tickets

### Agents (9 — TODOS internalizados no plugin)

```
agents/
  security-specialist.md       (opus)   — seguranca, OWASP, IDOR, auth
  backend-specialist.md        (sonnet) — performance, N+1, queries, API
  frontend-specialist.md       (sonnet) — WCAG, Blade, Bootstrap, Alpine.js
  quality-specialist.md        (sonnet) — PSR-12, SOLID, dead code, types
  qa-specialist.md             (sonnet) — unit tests, e2e, browser testing
  investigation-specialist.md  (opus)   — planning, validacao de findings
  clickup-manager.md           (sonnet) — TODAS as operacoes ClickUp API
  devils-advocate.md           (opus)   — review/filtering (JA EXISTE)
  code-simplifier.md           (sonnet) — complexidade (JA EXISTE)
```

**Carbon copy:** Cada agent internalizado recebe preamble de expertise que replica o contexto dos tipos built-in da Anthropic, mais as regras especificas do plugin.

**Prompts Protocol-Driven (OBRIGATORIO):**
Todos os agent `.md` files seguem o principio de **procedimento rigido com valvula de escape:**
1. **Passos numerados** — cada modo tem procedimento exacto, passo a passo
2. **Criterios de avaliacao explicitos** — nao "analisa qualidade", mas "verifica: type hints, return types, PSR-12, casts em models"
3. **Sem imposicao de ferramentas** — diz "procura por X no codebase", NAO "usa Grep". Agente escolhe a ferramenta
4. **Edge case thinking OBRIGATORIO** — em CADA passo, agente deve considerar edge cases e documenta-los
5. **Colaboracao entre agentes ENCORAJADA** — agentes DEVEM sugerir ideias/melhorias uns aos outros via SendMessage quando detectam oportunidades fora da sua area
6. **Desvios controlados** — se agente precisa sair do protocolo, DEVE pedir permissao ao Maestro com justificacao ANTES de agir
7. **Output format RIGIDO** — templates exactos por modo, sem campos extra, sem reordenacao
8. **FORBIDDEN list explicita** — cada agente tem lista do que NAO pode fazer

**Principio:** O protocolo dita O QUE fazer e COMO avaliar. O agente decide COMO executar (ferramentas, ordem de pesquisa). Desvios so com autorizacao.

**Persistencia & Memoria:**
- Agentes PERSISTEM entre sessoes — shutdown SO com instrucao explicita do USER
- Agentes sao criados via TeamCreate na primeira invocacao de qualquer skill
- Skills seguintes fazem resume dos agentes existentes (preserva contexto completo)

**Resiliencia — Modelo 3-Camadas + Heartbeat:**

Nota arquitectural: Subagentes (spawned via Agent tool) NAO recebem hooks do plugin.
O PreCompact hook so dispara na conversa principal (Maestro). Subagentes que esgotam
contexto simplesmente retornam a ultima mensagem — capturados via SubagentStop hook.

**Camada 1 — `/tmp/agent-{name}-progress.md` (efemero, safety net):**
  - QUEM escreve: Specialist DIRECTAMENTE (instrucao no system prompt)
  - QUANDO: event-based, apos cada PASSO significativo do procedimento
  - GRANULARIDADE durante FIX IMPLEMENTING: **per-file** (nao per-task)
    ```
    14:12:00 | IMPLEMENTING | ticket 86c8qhcfx | START | plan: 3 files
    14:12:15 | IMPLEMENTING | ticket 86c8qhcfx | MODIFIED | app/Http/Controllers/ImportController.php | +15 -3
    14:13:02 | IMPLEMENTING | ticket 86c8qhcfx | MODIFIED | app/Services/ImportService.php | +22 -8
    14:13:30 | IMPLEMENTING | ticket 86c8qhcfx | MODIFIED | resources/views/import/results.blade.php | +5 -2
    14:13:31 | IMPLEMENTING | ticket 86c8qhcfx | COMPLETE | 3/3 files
    ```
  - GRANULARIDADE durante AUDIT: 1 write por finding submetido ao DA
  - GRANULARIDADE durante PLANNING: 1 write por plano submetido + 1 por verdict recebido
  - DURACAO: efemero (sessao actual, perde-se se maquina reinicia)
  - PROPOSITO: "caixa negra de voo" — so lido se agente morre mid-task

**Camada 2 — `.claude/code-reviews/{dir}/{area}/{task_id}.md` (permanente, SOT local):**
  - QUEM escreve: APENAS o ClickUp Manager (via instrucao do Maestro)
  - QUANDO: apos cada ACCAO COMPLETA (commit, skip, rollback, status change)
  - CONTEUDO: frontmatter YAML (status, commit_sha, branch, last_synced) + body completo
  - Body progressivamente construido por cada skill:
    - Audit: cria ficheiro com Problema, Impacto, Evidencia, Correcao Sugerida, Como Testar
    - Planning: ADICIONA seccao `#### Planeamento` (agente, QA, ficheiros, wave, estimativa)
    - Fix: ADICIONA seccao `#### Commit` (SHA, branch, data, ficheiros)
  - DURACAO: permanente (cross-sessao, sobrevive a crashes, fim de sessao, reinicio)
  - PROPOSITO: source of truth local — contem TODA a historia do ticket

**Camada 3 — ClickUp API (ticket description, permanente, cloud):**
  - QUEM escreve: APENAS o ClickUp Manager (PUT markdown_description)
  - QUANDO: apos actualizar .md local (sync imediato, ou deferred se API down)
  - CONTEUDO: body do .md local SEM frontmatter (strip via awk antes do PUT)
  - O ticket no ClickUp ve TODAS as seccoes acumuladas (Problema → ... → Commit)
  - DURACAO: permanente (cloud, visivel a toda a equipa)
  - PROPOSITO: visibilidade externa — ClickUp UI

**Fluxo de dados (NUNCA shortcutted):**
```
/tmp/ (efemero) → Maestro le → instrui ClickUp Manager → .md local (SOT) → ClickUp (sync)
```
O `/tmp/` NUNCA vai directamente ao ClickUp. Sempre passa pelo Maestro + ClickUp Manager.

**Plugin Hooks (afectam Maestro, NAO subagentes):**
  - `PreCompact` — dispara quando contexto do Maestro atinge ~95%
    - Prompt-based hook: instrui Maestro a gravar estado para `/tmp/maestro-precompact-state.md`
    - Conteudo: wave actual, tickets processados, tickets pendentes, agentes activos, staged changes
    - Apos compactacao, Maestro le este ficheiro para recuperar contexto perdido
  - `SubagentStop` — dispara quando qualquer subagente termina (normal ou crash)
    - Maestro identifica qual agente morreu
    - Le `/tmp/agent-{name}-progress.md` para estado final
    - Decide: re-spawnar, continuar sem ele, ou escalar ao user
  - `Stop` — dispara quando sessao principal termina
    - Cleanup de ficheiros /tmp/ obsoletos (opcional)

**Heartbeat (Maestro → Subagentes):**
  - Frequencia: ~2min entre mensagens (ping ou keepalive)
  - Timeout de morte: 3min sem resposta a um ping directo
  - Protocolo apos deteccao de morte:
    1. Le `/tmp/agent-{name}-progress.md` — ultimo estado conhecido
    2. Verifica `git status` — ha staged changes orfas?
    3. Verifica `.md` local — ultimo estado sincronizado
    4. Decisao por estado:
       - Se STAGED sem DA review → re-submeter diff ao DA (novo DA instance ou existente)
       - Se IMPLEMENTING mid-file → `git checkout -- {files}` (unstage parcial), re-spawnar specialist
       - Se AWAITING_DA → verificar se DA respondeu (pode ter ficado em buffer)
       - Se COMPLETE (reportou ao Maestro) → nada a fazer, Maestro ja tem a info
    5. Re-spawnar specialist com contexto recuperado (ficheiro .md + ultimo progresso)
  - Regra para agentes: Se >3min sem mensagem do Maestro E tarefa concluida → gravar estado, pausar, esperar

### Separacao de Responsabilidades

```
MAESTRO (opus, o utilizador do Claude Code)
  - Orquestra o workflow
  - Decide proximos passos
  - Comunica com o user
  - Spawna/resume agents
  - FAZ COMMITS (unica accao de escrita)
  - NAO toca ClickUp API (delega ao ClickUp Manager)
  - NAO implementa fixes (delega a specialists)
  - NAO analisa codigo (delega a specialists)

CLICKUP MANAGER (sonnet, plugin agent)
  - PONTE TOTAL entre todos os agentes e o ClickUp
  - TODAS as operacoes ClickUp API sem excepcao:
    - Cria tickets (POST)
    - Actualiza tickets (PUT descricao, status, prioridade)
    - Apaga/modifica seccoes da descricao
    - Le/escreve comentarios
    - Muda status (COM VERIFICACAO DE EVIDENCIA)
    - Fecha tickets
  - Gere local cache (.claude/code-reviews/)
  - RECUSA mudar status sem evidencia
  - NENHUM outro agente toca na API do ClickUp — tudo passa por ele

SPECIALISTS (5 agents, plugin agents)
  - Analisam codigo (audit)
  - PLANEIAM os seus proprios tickets (planning) — cada specialist planeia a sua area
  - Implementam fixes (fix)
  - Reportam ao DA (NUNCA ao Maestro directamente)

INVESTIGATION SPECIALIST (opus, plugin agent)
  - NAO planeia tickets individuais (specialists fazem isso)
  - ANALISA dependencias cross-area, conflitos de ficheiros, ordenacao de waves
  - Trabalha par-a-par com DA (ping-pong) para validar wave plan
  - Recebe todos os planos dos specialists, faz meta-organizacao

DEVILS ADVOCATE (opus, plugin agent)
  - Valida TUDO: findings, plans, code, test results
  - 4 modos: FINDING-FILTER, PLANNING-REVIEW, CODE-REVIEW, QA-REVIEW (NOVO)
  - Unico gate entre specialists e Maestro
  - Verdicts: APPROVED / REJECTED / REQUEST-CHANGES
  - No planning: valida planos individuais (PLANNING-REVIEW) + ping-pong com Investigation

QA SPECIALIST (sonnet, plugin agent)
  - DUPLO PAPEL:
    - Fix skill: corre test suite (PHPUnit/Pest) apos CADA wave (sanity check, detectar regressoes)
    - Testing skill: testes browser via Chrome DevTools MCP (CRUD, interaccoes, console/network errors)
  - Testes BROWSER sao EXCLUSIVOS do testing skill (Chrome DevTools MCP)
  - Testes UNIT SUITE sao usados em fix (apos wave) E testing (baseline)
  - Screenshots como ferramenta de trabalho, APAGAR antes do shutdown
  - Reporta resultados ao DA para validacao

CLICKUP MANAGER — Evidence Gates (NOVO)
  - ready for dev -> in progress: EXIGE .md local (audit, MINIMO) + seccao Planeamento (planning, IDEAL). Warn sem planning
  - in progress -> code review: EXIGE staged diff + diff enviado ao DA via SendMessage
  - code review -> testing: EXIGE DA CODE-REVIEW APPROVED + commit SHA verificado + Decisoes Fix documentadas
  - code review -> in progress: DA REQUEST-CHANGES (specialist corrige, max 2 rounds)
  - testing -> deploy to staging: EXIGE DA QA-APPROVED verdict
  - testing -> ready for dev: QA fail MINOR (quick-fix, DA classifica)
  - testing -> planning: QA fail MODERATE/CRITICAL (re-investigacao, DA classifica)
  - Bloqueio em cascata: usa API nativa ClickUp dependencies (POST /task/{id}/dependency)
  - Qualquer transicao: EXIGE branch name no ticket
```

```
User invoca /clickup-code-review:planning
         |
    Maestro: fetch tickets "open" via ClickUp Manager
         |
    ClickUp Manager: retorna lista de tickets + comment check
         |
    FASE 1 — Planeamento por area (PARALELO, TRIANGULO):
      |
      [PARALELO] Cada specialist planeia os seus tickets:
        Security Specialist: planeia tickets de seguranca
        Backend Specialist: planeia tickets de performance
        Frontend Specialist: planeia tickets de WCAG/UI
        Quality Specialist: planeia tickets de qualidade
        QA Specialist: planeia tickets de testes

         |

        DA: desafia abordagem ("Porque A e nao B? Consideraste C?")

                       avalia impacto cross-area ("este fix afecta modulo X que nao foi considerado")
         |
      Ambos dao feedback ao specialist. Consensus dos 3 para aprovar:
        Se DA VALID + Investigation VALID: plano aprovado
        Se DA ou Investigation INVALID: specialist revisa ou ticket fechado
        Se DA NEEDS-CHANGE: specialist revisa abordagem (max 2 rounds)
        Se Investigation detecta falso positivo: DA arbitra → Maestro/ClickUp Manager fecha se confirmado
         |
    FASE 2 — Meta-organizacao (Investigation + DA ping-pong):
      |
      Investigation Specialist (opus): recebe TODOS os planos validados
         |
      Investigation analisa:
        - Dependencias cross-area (ticket Security depende de fix Quality?)
        - Conflitos de ficheiros (2 tickets modificam mesmo controller?)
        - Ordem de execucao (migrations antes de code, etc.)
        - Agrupamento em waves (sem conflitos intra-wave)
        - Routing: qual specialist fixa cada ticket
         |
      Investigation -> propoe wave plan ao DA
         |
      DA: desafia wave plan (ping-pong iterativo)
        - "Ticket A e B partilham ficheiro X, confirma conflito?"
        - Investigation investiga, responde
        - Ciclo ate consenso (max 3 rounds)
         |
      Investigation envia wave plan final ao Maestro
         |
    FASE 3 — Actualizacao:
      |
      Maestro -> ClickUp Manager: actualiza todos os tickets
        - Status "planning" -> "ready for dev"
        - Adiciona seccao #### Planeamento a cada ticket
        - Escreve ficheiro local .md com metadata + plano
         |
      Maestro: apresenta wave plan ao user
```

**Mudanca critica vs v4:** Planning DESCENTRALIZADO — cada specialist planeia a sua area em paralelo. Investigation + DA fazem meta-organizacao (waves, dependencias). Muito mais rapido.

## Workflow: /clickup-code-review:fix

**NOTA:** Testes BROWSER (Chrome DevTools MCP) NAO acontecem no fix skill — sao exclusivos do testing skill. O fix skill foca-se em implementar + DA CODE-REVIEW + commit + evidence gate. **O QA Specialist corre a test suite (PHPUnit/Pest) apos CADA wave** para detectar regressoes, mas sem testes browser.

```
User invoca /clickup-code-review:fix
         |
    Maestro: fetch tickets "ready for dev" via ClickUp Manager
         |
    ClickUp Manager: retorna tickets + comment check per ticket
         |
    Para CADA ticket na wave:
      |
      [Step 0] ClickUp Manager: comment detection
        Se breaking -> SKIP, queue re-planning
        Se non-breaking -> adaptar contexto
      |
      [Step 1] ClickUp Manager: status -> "in progress"
      |
      [Step 2] Maestro -> spawn Specialist (routing por area)
        Specialist recebe: ficheiro .md local (memoria completa do ticket)
      |
      [Step 3] Specialist (MODE: FIX): implementa fix + git add (NAO commit)
        Specialist escreve progresso em /tmp/agent-{name}-progress.md
      |
      [Step 4] Specialist -> SendMessage ao DA (CODE-REVIEW)
        Inclui: ticket context + staged diff (template obrigatorio)
        ClickUp Manager: status -> "code review" (evidencia: diff staged + enviado ao DA)
      |
      [Step 5] DA: APPROVED / REQUEST-CHANGES
        Se REQUEST-CHANGES:
          ClickUp Manager: status -> "in progress" (specialist corrige)
          specialist corrige na hora, re-stage, novo diff ao DA
          ClickUp Manager: status -> "code review" (novo diff enviado)
          (max 2 rounds, cada transicao documentada em #### Decisoes Fix)
        Apos 2 rejeicoes: escalate ao Maestro
      |
      [Step 6] Maestro COMMIT (unico momento de commit, SO se DA APPROVED):
        git commit -m "fix: {descricao}

        Ticket: {clickup_task_id}
        Area: {area}"
      |
      [Step 7] EVIDENCE GATE — ClickUp Manager:
        7a. Verifica commit existe: git log -1 --grep={ticket_id}
        7b. Extrai commit SHA + branch name
        7c. Consolida /tmp/agent-{name}-progress.md → .md local:
            Adiciona seccao #### Fix Log (specialist, ficheiros, DA verdict, timestamps)
            Adiciona seccao #### Decisoes Fix (cada round DA, cada feedback, cada decisao)
            Adiciona seccao #### Commit (SHA, branch, data, stats)
        7d. Actualiza frontmatter: commit_sha, branch, status, fix_attempts, last_synced
        7e. PUT body completo ao ClickUp (strip frontmatter, sync .md local → ClickUp)
        7f. SO ENTAO muda status "code review" -> "testing" (evidence gate: DA APPROVED + commit SHA + Decisoes Fix)
        7g. Se commit NAO encontrado -> RECUSA mudar status, alerta Maestro
        7h. Se #### Decisoes Fix ausente -> RECUSA mudar status ("Eventos nao documentados")
      |
    Wave completa -> Maestro: git status
                    QA Specialist: corre test suite (sail artisan test) — detectar regressoes
                    Se novos testes falham -> parar, reportar ao Maestro
                    Se OK -> next wave
    Tickets ficam em "testing" para o /testing skill validar
```

**Mudancas criticas vs v4:**
- Step 7: EVIDENCE GATE — commit SHA verificado ANTES de mudar status
- Step 7c: Branch + SHA escritos na descricao do ticket
- SEM testes browser no fix skill — testes browser exclusivos do testing skill. QA Specialist corre test suite (PHPUnit/Pest) apos cada wave para detectar regressoes
- Specialist corrige na hora se DA rejeita (sem revert)
- ClickUp Manager recusa mudar status sem evidencia
- .md local e REPOSITORIO DE MEMORIA COMPLETO — seccoes progressivas por fase (Fix Log, Commit, QA Results)
- /tmp/ consolidado no .md local no termino de cada fase, depois descartavel

### Interleaving Strategy (optimizacao ~2x throughput)

**Principio:** Specialists de areas DIFERENTES podem READ+IMPLEMENT em paralelo enquanto o DA revê outro ticket. Apenas STAGE+DA+COMMIT é sequencial (staging area global do git).

**Cronologia com 4 tickets (S1, S2 security + B1, B2 backend):**
```
t=0    Security: [S1 read+implement]
       Backend:  [B1 read+implement]           ← PARALELO (não toca git)
t=10   Security: [S1 stage]                    ← staging area ocupada
       DA:       [S1 review]
       Backend:  [B1 pronto, espera staging]
t=15   Maestro:  [S1 commit]                   ← staging area livre
       Backend:  [B1 stage]                    ← imediatamente após commit
       Security: [S2 read+implement]           ← começa próximo ticket
       DA:       [B1 review]
t=20   Maestro:  [B1 commit]
       Security: [S2 stage]
       Backend:  [B2 read+implement]
       DA:       [S2 review]
t=25   Maestro:  [S2 commit]
       Backend:  [B2 stage]
       DA:       [B2 review]
t=30   Maestro:  [B2 commit]

Total: ~30 unidades (vs ~63 sequencial puro)
```

**Cenário com REJEIÇÃO (DA rejeita S1):**
```
t=0    Security: [S1 read+implement]
       Backend:  [B1 read+implement]
t=10   Security: [S1 stage] → DA: [S1 review → REQUEST-CHANGES]
t=15   Security: [S1 fix based on feedback] → [re-stage]
       Backend:  [B1 espera staging] (Security re-staged, staging ocupada)
       DA:       [S1v2 review → APPROVED]
t=20   Maestro:  [S1 commit]                   ← staging livre
       Backend:  [B1 stage]
       DA:       [B1 review]
       Security: [S2 read+implement]
t=25   Maestro:  [B1 commit] ... continua normalmente

Impacto da rejeição: ~5 unidades extra (afecta Backend que esperava staging)
```

**Regras para o Maestro:**
1. **Distribuir por duracoes diferentes (anti-race-condition):** Ao atribuir tickets a specialists no inicio da wave, escolher tickets com estimativas DIFERENTES para cada specialist. Objectivo: specialists terminam em tempos escalonados naturalmente → nao criam fila simultanea no staging + DA.
   - **Mau:** Security 30m, Backend 30m, Frontend 30m → todos terminam a t=30 → race condition
   - **Bom:** Security 15m, Backend 30m, Frontend 45m → terminam a t=15, t=30, t=45 → pipeline fluido
   - Todos os specialists podem arrancar ao mesmo tempo — o escalonamento e por DURACAO dos tickets, nao por tempo de arranque
   - Se estimativas nao disponiveis: usar heuristica por severidade (Low/Medium geralmente mais rapidos)
2. **Pre-dispatch:** Quando um specialist termina e entra no ciclo stage→DA→commit, Maestro envia-lhe o proximo ticket para read+implement enquanto espera staging ficar livre
3. **Staging priority:** Se 2 specialists prontos para stage simultaneamente, prioridade por severidade do ticket
4. **Rejection handling:** Se DA rejeita, specialist fixa imediatamente (status volta a "in progress"). O proximo specialist espera staging ficar livre (poucos minutos de atraso, nao bloqueio total)

## Workflow: /clickup-code-review:testing (NOVO)

```
User invoca /clickup-code-review:testing
         |
    Maestro: spawn ClickUp Manager + QA Specialist + DA
         |
    ClickUp Manager: config check (token, list ID, shortname, status mapping)
         |
    [Opcao A] Teste completo da aplicacao (smoke + CRUD + interaccoes)
    [Opcao B] Verificacao de tickets em status "testing" (pos-fix)
         |
    QA Specialist (MODE: TESTING):
      - Login via Chrome DevTools MCP
      - Para cada pagina/ticket:
        1. navigate_page -> URL
        2. list_console_messages -> verificar erros JS
        3. list_network_requests -> verificar 4xx/5xx
        4. Interaccoes: click, fill, press_key
        5. Verificar CRUD completo onde aplicavel
        6. Screenshots como ferramenta de trabalho, APAGAR antes do shutdown
      |
    PARA TICKETS EM "TESTING" (Opcao B — validacao pos-fix):
      |
      QA Specialist -> SendMessage ao DA (QA-REVIEW)
        Inclui: ticket context + test evidence (URLs, accoes, resultados)
         |
      DA: QA-APPROVED / QA-REJECTED
         |
      Se QA-APPROVED:
        ClickUp Manager: status -> "deploy to staging"
        ClickUp Manager: actualiza .md local com resultados QA
         |
      Se QA-REJECTED:
        DA classifica gravidade: MINOR / MODERATE / CRITICAL
        |
        MINOR (quick-fix):

        BLOQUEIO EM CASCATA:

      |

         |

## Evidence Gates — Resumo

| Transicao | Evidencia Exigida | Quem Verifica |
|-----------|-------------------|---------------|
| open -> planning | Specialist assessment validado | DA (PLANNING-REVIEW) |
| planning -> ready for dev | DA VALID verdict + wave plan aprovado | ClickUp Manager |
| ready for dev -> in progress | MINIMO: .md local existe (audit). IDEAL: `#### Planeamento` presente (planning). Warn se sem planning | ClickUp Manager |
| in progress -> code review | Specialist staged diff + SendMessage ao DA (diff enviado) | ClickUp Manager |
| code review -> testing | DA CODE-REVIEW APPROVED + Commit SHA verificado + `#### Decisoes Fix` documentado | ClickUp Manager |
| code review -> in progress | DA REQUEST-CHANGES (specialist corrige, max 2 rounds) | ClickUp Manager |
| testing -> deploy to staging | DA QA-APPROVED verdict | ClickUp Manager |
| testing -> ready for dev | DA QA-REJECTED (MINOR) | ClickUp Manager |
| testing -> planning | DA QA-REJECTED (MODERATE/CRITICAL) | ClickUp Manager |
| * -> Closed | DA INVALID verdict (planning) | ClickUp Manager |

**Regra de ouro:** O ClickUp Manager RECUSA qualquer transicao sem a evidencia correspondente. Se um agente pede mudanca de status sem evidencia, o ClickUp Manager reporta ao Maestro em vez de executar.

**Sem RE-OPEN:** Tickets que falham QA voltam a "ready for dev" (MINOR) ou "planning" (MODERATE/CRITICAL). Nao ha status RE-OPEN — evita confusao com tickets do audit.

**Bloqueio em cascata:** ClickUp Manager usa API nativa de dependencies para bloquear tickets dependentes quando um ticket falha QA.

**OPERACAO: RATE LIMITING — Procedimento:**
```
PASSO 1: CONTADOR PROACTIVO
  - Manter contador de requests na sessao (variavel em memoria)
  - Threshold: 80 requests por minuto
  - Janela: rolling 60 segundos

PASSO 2: PRE-REQUEST CHECK
  - Antes de CADA request ao ClickUp:
  - Se contador >= 80 no ultimo minuto:
    → sleep 20s
    → Log: "Rate limit proactivo: {counter} requests no ultimo minuto, pausa 20s"
    → Durante espera: trabalhar em operacoes locais (.md files)

PASSO 3: HANDLE 429
  - Se ClickUp retorna HTTP 429:
    → wait 60s
    → retry 1x
    → Se 2o 429 → modo offline (trabalhar so no .md local)
    → Log: "Rate limit 429: espera 60s, retry"

PASSO 4: REPORTAR
  - Se 429 ou proactivo triggered → avisar Maestro
  - Sugerir ao Maestro agrupar operacoes se possivel
```

**Error Handling API:**
- Rate limit (429): espera 60s, retry 1x. Durante a espera, continua a trabalhar no .md local (previne bottleneck)
- Timeout/502: retry 1x apos 10s. Durante a espera, anota tarefas pendentes no .md local
- Apos 2 falhas consecutivas: reporta ao Maestro, continua em modo offline (trabalha so no .md local)
- Modo offline: o .md local e a SOT. Sincroniza com ClickUp quando API voltar
- Progresso NUNCA se perde — .md local tem sempre o estado mais recente

**O que o ClickUp Manager NAO faz:**
- NAO analisa codigo
- NAO implementa fixes
- NAO toma decisoes de orquestracao
- NAO comunica com o user directamente
- NAO avalia findings ou diffs

**Credential Security Enforcement:**
- Antes de cada PUT descricao ou POST comment, scan para patterns:
  - `pk_[a-zA-Z0-9]{8,}` (ClickUp tokens)
  - `sk_[a-zA-Z0-9]{8,}` (Stripe-like keys)
  - `password\s*[:=]\s*\S+` (inline passwords)
  - `Bearer [a-zA-Z0-9\-._~+/]+=*` (auth tokens)
- Se detectar → REJEITA operacao, reporta ao Maestro

  3.3 Deep Nesting:
    - if/else com 3+ niveis de aninhamento
    - Procurar patterns de early return ausentes
    - Edge case: switch/case vs strategy pattern
  3.4 Cyclomatic Complexity:
    - Metodos com >5 caminhos de decisao (if/else/switch/ternary)
    - Procurar metodos com muitos parametros (>4) — indicador de responsabilidade multipla
    - Edge case: metodos de validacao (naturalmente complexos, avaliar se extraccao faz sentido)
  3.5 Duplicacao:
    - Procurar blocos de codigo repetidos em 2+ locais
    - Procurar patterns copy-paste (mesma estrutura, variaveis diferentes)
    - Edge case: duplicacao aceitavel (boilerplate framework) vs duplicacao problematica
  3.6 Boolean Flags:
    - Metodos com parametros boolean que mudam comportamento
    - Ternarios encadeados dificeis de ler
    - Edge case: feature flags (aceitavel) vs control flow booleans (problematico)

  3.7 ANALISE LIVRE (OBRIGATORIO):
    - AGRUPA findings relacionados (3 controllers com o mesmo problema = 1 finding)
    - Se encontrar algo fora de complexidade → SendMessage ao specialist da area

PASSO 4: REPORTAR (template standard, 1 finding por SendMessage ao DA)
  - Append /tmp/: "{timestamp} | AUDIT | finding {titulo} | SUBMITTED_TO_DA"
  - Apos DA verdict: append /tmp/: "{timestamp} | AUDIT | finding {titulo} | {APPROVED/REJECTED}"
PASSO 5: FINALIZAR
```

- Foco: worst offenders (20% do codigo que causa 80% da dor)
- Routing: SendMessage ao DA (FINDING-FILTER)
- **Heranca de v4:** Mantém logica do agent-prompt `code-simplifier.md`

**MODE: FIX (existente — manter)**
- Revisa staged diffs para over-engineering, abstraccoes desnecessarias, scope creep
- Output: Ticket ID, Verdict (CLEAN/SIMPLIFY), Complexity assessment, Issues, Suggestions
- Foco: apenas no diff, nao no codigo envolvente

**Tickets de complexidade no PLANNING:** Investigation Specialist planeia tickets de complexidade (area "complexity") durante a meta-organizacao. O Investigation tem contexto cross-area ideal para avaliar se uma simplificacao e REALMENTE benefica. Regra: "Se nao esta partido, nao arranjes." Simplificacoes so sao aprovadas se trazem beneficios concretos de manutencao futura E leitura (para humanos E AI). Areas criticas do sistema exigem planeamento hiper-profundo pelo Investigation + DA triangulo reforçado.

**Mudanca critica vs v4:** Specialists agora participam em 3 fases (audit + planning + fix) em vez de apenas 1.
Investigation Specialist faz meta-organizacao (waves) em vez de planning individual.
Investigation Specialist TAMBEM planeia tickets de complexidade (avaliacao de risco/beneficio cross-area).
QA Specialist EXCLUSIVAMENTE no testing skill (nao no fix).

**Nota:** "—" significa que o agente nao e usado nessa fase. O Maestro pode consultar qualquer agente se necessario.

## Regras de Comunicacao Inter-Agentes

**Principio: Comunicacao ABERTA com Maestro SEMPRE informado.**

### Quem pode falar com quem
- **Qualquer agente** pode falar com **qualquer outro** via SendMessage
- Colaboracao entre specialists e ENCORAJADA (sugestoes cross-area)
- DA pode pedir clarificacoes directamente a qualquer specialist
- Investigation pode consultar qualquer specialist durante wave planning

### Regras OBRIGATORIAS (3 guardrails)
1. **Maestro recebe copia de TODAS as mensagens criticas:**
   - Verdicts do DA (APPROVED, REJECTED, REQUEST-CHANGES, QA-APPROVED, QA-REJECTED)
   - Escalacoes (apos 2 rounds sem consenso)
   - Bloqueios (agente nao consegue prosseguir)
   - Mudancas de scope (agente quer sair do protocolo)
2. **SO o Maestro fala com o User** — nenhum agente comunica directamente com o utilizador
3. **SO o ClickUp Manager toca na API do ClickUp** — nenhum outro agente faz requests HTTP ao ClickUp

### Formato de mensagens SendMessage
Todas as mensagens entre agentes DEVEM incluir:
- **De:** {agent-name}
- **Para:** {agent-name}
- **Tipo:** FINDING / VERDICT / SUGGESTION / QUESTION / ESCALATION / REPORT
- **Conteudo:** {mensagem}

## Error Handling Global

### Agent Death Recovery

**Deteccao:** Heartbeat ping cada ~2min. Timeout de morte: 3min sem resposta. Tambem: `SubagentStop` hook (deteccao imediata se agente retorna/crash).

**Protocolo de recuperacao:**
1. Le `/tmp/agent-{name}-progress.md` — per-file granularity, ultimo estado conhecido
2. Le `.md` local do ticket via ClickUp Manager — memoria completa cross-sessao
3. Verifica `git status` — staged changes orfas? Ficheiros parcialmente modificados?
4. Decisao por estado (ver tabela em "Resiliencia — Modelo 3-Camadas + Heartbeat"):
   - STAGED sem DA review → re-submeter diff ao DA
   - IMPLEMENTING mid-file → `git checkout -- {files}`, re-spawnar specialist com "continuar do ficheiro N"
   - AWAITING_DA → verificar buffer do DA (pode ja ter respondido)
   - COMPLETE → nada a fazer
5. Re-spawnar specialist com contexto: ficheiro `.md` local + ultimo progresso de /tmp/
6. Novo agente continua de onde o anterior parou

### Git Conflicts

**Resolucao:** Maestro resolve, com ajuda do DA se necessario.

**Protocolo:**
1. Maestro detecta conflito (git status mostra CONFLICT)
2. Maestro analisa ficheiros em conflito
3. Se conflito simples (whitespace, imports): Maestro resolve directamente
4. Se conflito complexo (logica, multi-ficheiro): Maestro consulta DA para decisao
5. Apos resolucao: `git add` ficheiros resolvidos, continuar wave

**Prevencao:** Wave conflict detection (set intersection) reduz 90%+ dos conflitos. Conflitos residuais sao por mudancas externas (outro developer, hotfix, etc.)

### ClickUp API Failures

**Ver seccao ClickUp Manager → Error Handling API** para detalhes completos.
- Rate limit (429): wait 60s, retry, trabalha offline no .md local
- Timeout/502: retry 1x, modo offline apos 2 falhas
- .md local e SEMPRE a SOT — sincroniza quando API voltar

### Specialist Cannot Implement Fix

1. Specialist reporta falha ao Maestro com motivo detalhado
2. Maestro avalia: problema no plano (re-plan) ou problema tecnico (skip)
3. Se skip: ClickUp Manager adiciona comentario com razao, ticket fica "in progress"
4. Se re-plan: ticket volta para "planning" para re-investigacao

### Multiple Agents Die Simultaneously

1. Maestro pausa wave actual
2. Verifica `/tmp/agent-*-progress.md` para TODOS os agentes
3. Respawna agentes por ordem de prioridade (DA primeiro, depois specialists)
4. Retoma wave do ponto de interrupcao

### Disk/Context Exhaustion

**Subagente esgota contexto:**
- Subagente retorna ultima mensagem ao Maestro (comportamento nativo do Agent tool)
- `SubagentStop` hook dispara → Maestro detecta morte
- Maestro segue protocolo heartbeat (ver "Resiliencia — Modelo 3-Camadas + Heartbeat")
- Recovery: `/tmp/agent-{name}-progress.md` (per-file granularity) + `.md` local

**Maestro esgota contexto (~95%):**
- `PreCompact` hook dispara (prompt-based)
- Maestro grava estado para `/tmp/maestro-precompact-state.md`:
  ```
  wave_actual: 2
  tickets_processados: [86c8qhcfx, 86c8qhcgd, 86c8qhch2]
  tickets_pendentes: [86c8qhchg, 86c8qhchx]
  ticket_em_progresso: 86c8qhchg (specialist: security, estado: AWAITING_DA)
  agentes_activos: [security-specialist, devils-advocate, clickup-manager]
  staged_changes: sim (ticket 86c8qhchg, 2 files)
  branch: fix/clickup-review-2026-03-10
  ```
- Apos compactacao, Maestro le este ficheiro + todos os `/tmp/agent-*-progress.md`

### Maestro Compactacao Durante Wave Activa

**Modelo hibrido:**
- Agentes **mid-task** (specialist a implementar fix, DA a rever diff) **CONTINUAM** a tarefa actual ate conclusao
- Agentes que **terminam** a tarefa actual **PAUSAM** — nao pegam na proxima tarefa sozinhos
- Cada agente grava progresso em `/tmp/agent-{name}-progress.md` (per-file granularity)
- Quando Maestro retoma apos compactacao:
  1. Le `/tmp/maestro-precompact-state.md` — recupera estado de orquestracao
  2. Le `/tmp/agent-*-progress.md` — verifica estado de cada agente
  3. Faz heartbeat ping a cada agente activo — confirma quais estao vivos
  4. Re-distribui tarefas pendentes aos agentes vivos
  5. Re-spawna agentes mortos com contexto recuperado

**Regra para agentes:** Se >3min sem mensagem do Maestro E tarefa actual concluida → gravar estado completo em /tmp/, pausar, esperar

## Agent Team Lifecycle

### Criacao de Agentes

Agentes sao criados **on-demand** na primeira invocacao de cada skill:

| Skill invocado | Agentes criados |
|----------------|----------------|
| `:audit` | Specialists seleccionados + DA + ClickUp Manager |
| `:planning` | Specialists seleccionados + DA + Investigation + ClickUp Manager |
| `:fix` | Specialists por area dos tickets + DA + ClickUp Manager |
| `:testing` | QA Specialist + DA + ClickUp Manager |
| `:setup` | Nenhum (Maestro faz tudo) |

**Regras:**
1. **TeamCreate na primeira invocacao** de qualquer skill que precise de agentes
2. **Resume nas invocacoes seguintes** — se o user invoca `:planning` depois de `:audit`, os agentes ja existentes sao resumidos (contexto preservado)
3. **Agentes nao criados em antecipacao** — so sao criados quando o skill os precisa. Se o user invoca `:fix` sem ter feito `:audit`, os specialists sao criados na hora com contexto do ticket .md local (sem historico de audit, mas funcional)
4. **Agentes persistem entre skills** — shutdown SO por instrucao explicita do USER ("desliga os agentes", "termina a sessao")
5. **Se agente morre** (context exhaustion, crash): re-spawnar via Agent tool com contexto recuperado de `/tmp/` + `.md` local

### Pool vs On-Demand

**NAO ha pool pre-criado.** Agentes sao criados on-demand porque:
- Cada skill usa um subset diferente de agentes
- Criar todos os 9 agentes upfront desperdicaria contexto dos que nao sao usados
- On-demand permite adaptar o contexto do agente ao skill actual

**Excepcao:** DA e ClickUp Manager sao transversais — criados no primeiro skill e resumidos em todos os seguintes.

## Testing Login Handling

### Sessao de Browser (QA Specialist)

**Login UMA VEZ, manter sessao:**
1. QA Specialist faz login no inicio do skill `:testing` (ou `:fix` VALIDATION mode)
2. Sessao mantida via cookies do Chrome DevTools MCP
3. Todas as paginas testadas dentro da mesma sessao

**Se sessao expira mid-test:**
1. QA Specialist detecta redirect para pagina de login (URL check apos navigate_page)
2. Re-login automatico com mesmas credenciais
3. Continuar teste da pagina actual (retry navigate_page)
4. Log: `"{timestamp} | SESSION_EXPIRED | re-login | {url_actual}"`

**Credenciais — ordem de deteccao:**
1. `.claude/credentials.local.md` — preferido (segue politica CLAUDE.md)
2. `.env` — fallback (APP_URL, DB credentials para verificar setup)
3. AskUserQuestion — ultimo recurso (so se ambos os ficheiros nao existem)

**Regra:** QA Specialist NUNCA inclui credenciais nos reports. Usa placeholders: `{TEST_EMAIL}`, `{TEST_PASSWORD}`.

**Multi-role testing (quando aplicavel):**
- Se ticket exige teste com diferentes roles (admin vs user): logout + login com role diferente
- Cada role usa credenciais do `.claude/credentials.local.md`
- Log: `"{timestamp} | ROLE_SWITCH | {from_role} -> {to_role}"`

## DA como Sombra Durante Implementacao

O Devil's Advocate (DA) actua como **reviewer em tempo real** durante a implementacao deste plano. Nao espera pelo fim — valida ficheiro a ficheiro, a medida que sao escritos.

### Protocolo

```
PARA CADA ficheiro implementado pelo Maestro:

1. Maestro termina de escrever/editar o ficheiro
2. Maestro envia ao DA via SendMessage:
   - Ficheiro: {path completo}
   - Fase: {1/2/3/4/5}
   - Seccao do plano: {referencia a seccao relevante do v5.0-architecture-plan.md}
   - Resumo das decisoes: {o que foi implementado e porque}

3. DA le o ficheiro implementado
4. DA compara com a seccao correspondente do plano:
   - O ficheiro implementa TUDO o que o plano especifica?
   - Ha desvios? Se sim, sao justificados?
   - Ha seccoes do plano OMITIDAS na implementacao?
   - A qualidade do output (templates, procedimentos, regras) esta ao nivel especificado?

5. DA emite verdict:
   - CONFORME: ficheiro alinhado com o plano
   - DESVIO-JUSTIFICADO: ficheiro diverge mas com razao valida (documentar no plano)
   - DESVIO-INJUSTIFICADO: ficheiro diverge sem justificacao → Maestro corrige
   - INCOMPLETO: ficheiro falta implementar partes do plano → Maestro completa

6. Se DESVIO-INJUSTIFICADO ou INCOMPLETO:
   - Maestro corrige o ficheiro
   - Re-submete ao DA (max 2 rounds por ficheiro)
   - Apos 2 rejeicoes → discussao com user
```

### Quando Spawnar o DA

- O DA e spawnado no INICIO da Fase 1 (antes do primeiro ficheiro)
- O DA recebe o plano completo (`v5.0-architecture-plan.md`) como contexto
- O DA PERSISTE durante toda a implementacao (todas as 5 fases)
- Se o DA morrer (context exhaustion): re-spawnar com plano + lista de ficheiros ja validados

### O Que o DA Verifica

| Aspecto | Verificacao |
|---------|------------|
| **Completude** | Todas as seccoes do plano estao implementadas no ficheiro? |
| **Fidelidade** | O conteudo corresponde ao especificado (templates, procedimentos, regras)? |
| **Consistencia** | O ficheiro e consistente com ficheiros ja validados? |
| **Qualidade** | Output formats, forbidden lists, communication rules — tudo presente? |
| **Linguagem** | PT-PT com acentos nos conteudos narrativos? Termos tecnicos em backticks? |

### Output Format do DA (Implementacao)

```markdown
### [CONFORME/DESVIO-JUSTIFICADO/DESVIO-INJUSTIFICADO/INCOMPLETO] — {filename}
- **Fase:** {1-5}
- **Seccao do plano:** {referencia}
- **Cobertura:** {X}% do especificado no plano
- **Desvios:** {lista ou 'nenhum'}
- **Omissoes:** {lista ou 'nenhuma'}
- **Notas:** {observacoes adicionais}
```

### Regra de Ouro

**NENHUM ficheiro avanca para a proxima fase sem verdict CONFORME ou DESVIO-JUSTIFICADO do DA.** Isto garante que a implementacao e fiel ao plano debatido e aprovado.
