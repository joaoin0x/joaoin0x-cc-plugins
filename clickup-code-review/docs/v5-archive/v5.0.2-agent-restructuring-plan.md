### Agent Architecture Pattern

**7 specialists** seguem padrão idêntico: Persona → MODE SELECTION → AUDIT (PASSO 1-7) → PLANNING (PASSO 1-5) → FIX (PASSO 1-8) → Communication → Forbidden → Shutdown.

## Plano de Reestruturação

### Princípio: Cada agent .md ≤ 10K chars. Conteúdo shared → skills injectados via spawn template.

### Regra Fundamental: ZERO INSTRUÇÕES ELIMINADAS

Este plano **rearranja** conteúdo — NÃO elimina instruções. Os agents ficam mais leves porque:
1. **22,750 chars de conteúdo duplicado** (idêntico em 8-9 agents) são consolidados em 3 shared skills
2. Parágrafos verbosos são **condensados em bullets** (mesma informação, menos palavras)
3. Exemplos longos inline são **encurtados** (1 exemplo representativo em vez de 3)

**O que NÃO se perde:** Domain-specific knowledge (OWASP checklists, WCAG criteria, query patterns, etc.), mode-specific logic, output templates, edge case handling. Se algo domain-specific existir apenas num agent, fica nesse agent.

### Fase 1: Criar 3 Shared Skills

Agents carregam via `skills:` no frontmatter. Cada skill tem frontmatter próprio.

**1.1 `skills/shared/pipeline-rules.md`** (~3K chars)

Conteúdo a extrair dos 9 agents:
- **Standard Finding Format** template (### SHORTNAME - Título...)
- **Communication rules:** SendMessage patterns, routing (specialist→DA, DA→Maestro, Maestro→CU Manager)
- **Streaming rule:** 1 finding por vez, nunca batch
- **Progress tracking:** timestamp | mode | action | detail format
- **Credential security:** NEVER include real credentials, usar placeholders
- **Forbidden actions:** (shared skeleton) NUNCA comunicar com user, NUNCA fazer commits, NUNCA shutdown sem ordem
- **DESVIO DO PROTOCOLO:** escalation clause
- **Shutdown rule:** só o Maestro envia shutdown_request

**1.2 `skills/shared/planning-protocol.md`** (~2.5K chars)

Conteúdo a extrair dos 7 agents com PLANNING mode:
- **PASSO skeleton:** 1-Contextualizar, 2-Reler código, 3-Validar finding, 4-Planear 2 abordagens, 5-Enviar plano
- **Triangle validation:** Specialist + DA + Investigation
- **Verdict types:** VALID, INVALID, PARTIAL, NEEDS-CHANGE (max 2 rounds)
- **Template `#### Planeamento`** COM sub-secções:
  - `##### Correcção Sugerida (Actualizado após Planeamento)`
  - `##### Como Testar (Actualizado após Planeamento)`
- **Audit sections IMMUTABLE** — regra explícita
- **Wave planning format** (waves, dependencies, conflicts)

**1.3 `skills/shared/fix-protocol.md`** (~2.5K chars)

Conteúdo a extrair dos 7 agents com FIX mode:
- **PASSO skeleton:** 1-Contexto, 2-Ler ficheiros, 3-Implementar, 4-Self-validate, 5-Stage, 6-Capture diff, 7-Enviar DA, 8-Aguardar verdict
- **Specialist↔DA direct flow** (Maestro NOT in loop)
- **Stage → diff → DA CODE-REVIEW → commit cycle**
- **Evidence gates** por status transition
- **Max 2 revision rounds** antes de escalation ao Maestro
- **Rollback:** `git revert <commit>`, max 2 retries
- **Forbidden:** NUNCA fazer commit, NUNCA push, NUNCA operações destrutivas

### Fase 3: Actualizar Frontmatter de Todos os Agents

**Template frontmatter (aplicar a todos):**

```yaml
---
name: {agent-name}
description: {manter existente — triggering conditions + examples}
model: {manter — opus|sonnet|inherit}
color: {manter existente}
tools: [Read, Grep, Glob, Bash, Write, Edit, SendMessage]
---
```

**Nota sobre campos:** Based on research of official Claude Code docs, `skills`, `maxTurns`, `memory`, and `permissionMode` are NOT documented frontmatter fields for plugin agents. The documented fields are: `name`, `description`, `model`, `color`, `tools`. Shared content must be injected via the spawn template in SKILL.md, not via undocumented frontmatter.

**Implicação:** Em vez de `skills: [pipeline-rules]` no frontmatter, o Maestro (via SKILL.md) deve incluir shared skill content no spawn template de cada agent. Alternativa: cada agent tem instrução "Read `skills/shared/pipeline-rules.md` at startup" no body.
